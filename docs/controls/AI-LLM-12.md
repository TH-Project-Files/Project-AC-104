# AI-LLM-12: Put strict safety checks in place when letting AI talk directly to your company databases (like translating plain text into SQL queries).

**Project:** Argus Centurion (AC-104)  
**Category:** Layer 7: Internal LLMs & Agentics  
**Implementation Group:** IG 2  
**Risk Level:** 3-Low  
**Framework Mappings:** CIS v8: `16.1` | NIST CSF: `PR.DS`

---

## Control Details
Detailed Description:
Implementing strict safety checks involves creating a layered defense to ensure that natural language prompts converted into SQL queries do not execute malicious commands, unauthorized data access, or destructive operations like dropping tables.

Why AI Compounds Risk:
AI exacerbates this risk by lowering the barrier to entry for attackers, allowing those without SQL knowledge to generate complex, database-specific injection payloads and bypass traditional security filters that look for manual SQL syntax.

Examples:
1. Use parameterized queries or prepared statements to ensure that any data generated by the LLM is treated strictly as a value rather than executable code.
2. Apply the principle of least privilege by connecting the AI to the database using a dedicated account with restricted, read-only access to specific views rather than raw base tables.
3. Implement human-in-the-loop or automated validation steps where generated SQL is inspected for high-risk keywords or structural anomalies before being executed on the live database.
